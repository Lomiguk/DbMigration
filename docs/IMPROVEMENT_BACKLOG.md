# Backlog технических улучшений

Документ фиксирует идеи развития, которые обсуждались после внедрения adaptive batch sizing. Это не список обязательных задач для ближайшего релиза, а инженерный backlog для последовательной проверки гипотез на тестовом стенде.

## 1. Автоматическая балансировка параллелизма

**Идея:** подбирать количество рабочих потоков не вручную, а по измеряемому соотношению времени ожидания и времени полезной работы.

**Зачем:** если приложение часто ждёт БД, соединения или I/O, умеренное увеличение параллелизма может повысить throughput. Если БД уже перегружена, рост числа потоков только усилит конкуренцию.

**Что измерять:**
- длительность batch;
- ожидание соединений из пула;
- активные и ожидающие соединения HikariCP;
- throughput строк/сек;
- долю времени в `source_read`, `copy_data`, `mapping_save`, `fk_lookup`.

**Как проверять:** сравнить фиксированный `maxPoolSize` с адаптивным режимом на одинаковом seed и объёме данных. Отдельно проверять малые, средние и большие наборы.

## 2. Порог включения adaptive-режимов

**Статус:** реализовано базовое правило: `adaptiveWarmupBatches=2` и `minAdaptiveRows=5000`. Практический прогон на `199860` строк: adaptive decisions начались с крупных таблиц (`users#5`), малые справочники не меняли batch; COPY занял `8205ms`.

**Идея:** включать adaptive batch не сразу или не для всех таблиц, а после минимального числа строк/батчей либо после первичной оценки скорости.

**Зачем:** на малых наборах adaptive-логика может не дать выигрыша или ухудшить результат из-за лишнего дробления batch. Это было видно на варианте с низким target `50ms`.

**Возможная эвристика:**
- не менять размер batch для таблиц меньше N строк;
- начинать адаптацию после первых 2-3 batch;
- не уменьшать batch из-за единичного всплеска, требовать повторения.

**Как проверять:** прогнать матрицу `10k / 30k / 100k+` и сравнить fixed batch, текущий adaptive и adaptive с порогом включения.

## 3. Адаптивное ограничение нагрузки по WAL lag

**Идея:** снижать нагрузку первичной миграции или фоновых операций, если replication slot начинает заметно отставать.

**Зачем:** при zero-downtime миграции важно не только быстро скопировать данные, но и не накопить слишком большую дельту WAL, которую потом долго догонять.

**Что измерять:**
- `replication_lag_bytes`;
- скорость чтения WAL;
- скорость применения WAL;
- размер неприменённой дельты;
- время `read_duration_ms` и `apply_duration_ms`.

**Возможная реакция системы:**
- уменьшить batch size;
- снизить параллелизм;
- вставлять паузы между batch;
- временно отдавать приоритет WAL apply.

**Как проверять:** сценарий `copy -> slot -> live writes -> replicate -> validate`, сравнивая общий downtime-risk и время догонки WAL.

## 4. Оптимизация WAL apply

**Статус:** реализован первый шаг: bulk-resolve mapping через `getNewIds()` для FK, UPDATE и DELETE внутри WAL batch. Практический WAL-прогон на дельте `19986` событий: `read_duration_ms=10612`, `apply_duration_ms=9165`, `events_applied=19986`, `events_failed=0`, validate `20/20`. Более глубокие оптимизации apply pipeline остаются в backlog.

**Идея:** ускорить применение WAL-событий в target.

**Зачем:** в полном тесте WAL read занял `10770ms`, а apply `28874ms`. После оптимизации COPY именно WAL apply становится заметным узким местом.

**Направления:**
- группировать события по таблицам;
- применять события batch-операциями;
- уменьшить число mapping lookup;
- использовать подготовленные statements повторно;
- отдельно оптимизировать INSERT/UPDATE/DELETE.

**Что измерять:**
- `events_read`;
- `events_applied`;
- `events_failed`;
- `read_duration_ms`;
- `apply_duration_ms`;
- latency применения по типам операций.

**Как проверять:** на фиксированной дельте WAL сравнить текущий apply и новый batch apply. Обязательная проверка: `validate` после применения.

## 5. Расширение performance-метрик

**Идея:** добавить метрики, которые помогают объяснять качество миграции точнее, чем только total time.

**Кандидаты:**
- p95/p99 batch duration;
- p95/p99 mapping lookup;
- wait time по connection pool;
- cache hit/miss по таблицам;
- размер COPY payload;
- rows/sec по фазам batch;
- число batch, где adaptive увеличил или уменьшил размер;
- WAL apply latency по типам событий.

**Зачем:** такие метрики позволяют отличить реальный выигрыш алгоритма от случайного шума окружения, прогретого page cache или особенностей конкретного seed.

**Как проверять:** метрики должны попадать в `performance_logs/run_*/` и быть пригодны для сравнения нескольких прогонов.

## 6. Автонастройка cacheLimit и HYBRID-порогов

**Идея:** подбирать параметры кэша на основе фактического FK-профиля и hit rate.

**Зачем:** HYBRID полезен, если маленькие таблицы действительно часто используются как parent-таблицы. Если pinned cache почти не используется, режим может не окупаться.

**Что измерять:**
- `hit_rate`;
- `miss_count`;
- `eviction_count`;
- `lazy_cache_size`;
- `pinned_cache_size`;
- DB lookup latency по таблицам.

**Возможные решения:**
- рекомендовать `cacheLimit` после пробного прогона;
- отключать HYBRID, если pinned-часть не даёт hit rate;
- менять долю памяти между pinned и lazy cache.

**Как проверять:** сравнить LAZY/HYBRID на одинаковом dataset, отдельно анализируя FK-heavy таблицы.

## 7. Стратегия для циклов зависимостей

**Идея:** обрабатывать циклические зависимости как отдельные группы таблиц, а не пытаться решить их обычной топологической сортировкой.

**Зачем:** `DirectedAcyclicGraph` корректно работает для ациклического графа. Если в схеме есть цикл FK, проблему нужно локализовать: показать группу таблиц, которая образует цикл, и применить отдельную стратегию миграции.

**Возможная стратегия:**
- определить strongly connected component;
- создать таблицы без циклических FK;
- заранее выделить новые BIGINT id;
- загрузить связанные строки группой;
- после загрузки создать внешние ключи и проверить целостность.

**Как проверять:** добавить тестовую схему с циклом `A -> B -> A`, затем проверить миграцию данных и создание FK после загрузки.

## 8. Production-строгость rollback

**Идея:** привести rollback к одному соединению и одной явной транзакции.

**Зачем:** концептуально rollback описывается как атомарный откат таблицы. Если часть операций выполняется через разные соединения, при сбое можно получить частично очищенную таблицу или несогласованное состояние mapping/state.

**Что улучшить:**
- `TRUNCATE target`;
- удаление `migration_mapping`;
- обновление `migration_state`;
- валидация rollback;
- запись ошибки.

Все эти действия должны быть в одной транзакции там, где это технически возможно.

**Как проверять:** integration-тест с искусственной ошибкой в середине rollback и проверкой, что состояние не стало частично изменённым.

## 9. Пересмотр Memory-Filtered Delta Sync

**Идея:** явно ограничить роль Memory-Filtered Delta Sync или заменить его WAL-сценарием там, где нужна полноценная синхронизация изменений.

**Зачем:** текущий memory-filtered проход добавляет новые строки, но не покрывает UPDATE и DELETE. При наличии WAL-репликации это решение спорное для сценариев, где требуется настоящая синхронизация изменений.

**Возможные варианты:**
- оставить `sync` только как догон новых INSERT;
- явно документировать, что `sync` не заменяет WAL;
- для production zero-downtime рекомендовать `replicate`;
- добавить предупреждение в CLI;
- в перспективе удалить дублирующий путь, если WAL покрывает нужный сценарий.

**Как проверять:** тесты на INSERT/UPDATE/DELETE после первичного COPY: `sync` должен честно показывать свои ограничения, а `replicate` должен доводить target до валидного состояния.
